# AgentPlan动态更新设计文档总览

## 动机

Flink Agents 作业通常是长期运行的。作业提交后，Agent 会持续消费业务事件，并依靠 keyed state、checkpoint 和 durable execution 保证一致性与恢复。

当前 `AgentPlan` 在作业提交时固定下来。`AgentPlan` 包含 action 编排、事件路由、prompt/resource/tool/model 配置。改一句 prompt、换一个 model、重定向一个 tool，或者调整 action 连接关系，都需要重新提交作业。

这对生产环境的 Agent 成本很高：

*   重启会中断正在处理的事件链。

*   变更链路长，策略调整不够快。

*   已有 keyed state 和 durable execution 上下文需要随作业恢复，操作风险高。

*   对 prompt、resource、tool/model 这类策略配置来说，重提作业太重。

本设计聚焦一个核心问题：**运行中的 AEO 如何在不重启作业的情况下，从旧 AgentPlan 原地切换到新 AgentPlan，并在失败恢复后仍然用对 plan。**

## 拆分方式与截面

整个问题在一个清晰的截面处一分为二：**"新 plan 如何送达每个 AEO subtask"** 和 **"subtask 拿到新 plan 后如何切换"**。截面本身就是一条消息加一个约定——coordinator 广播的 `PlanUpdateEvent(planVersion, planId, canonicalPlanJson)`，以及 subtask 侧"收到后只做不进入 CPython 的静态校验、成为 pendingPlan；静态 prepare 成功后等待下一次 checkpoint barrier 激活"的处理契约。截面之前几乎全是 Flink 原生组件（REST、OperatorCoordinator、SubtaskGateway），与 Agent 语义无关；截面之后全是 AEO 内部的事，与传输通道无关。两侧可以独立设计、独立演进，因此各成一篇：

*   **[Part.1 AgentPlan更新链路设计](<Part.1 AgentPlan更新链路设计.md>)** —— 外部链路：用户手里的一份新 AgentPlan JSON，如何变成每个 subtask 内存中的 pendingPlan。覆盖通道选型（为什么是 OperatorCoordinator 广播而不是重提作业/broadcast stream/外部存储轮询）、基于 agentName 的定向寻址、Java 客户端与 Python facade、投递与定序语义、"接受 ≠ 生效"的响应语义、分层校验、失败点走查。

*   **[Part.2 数据流切换 AgentPlan](<Part.2 切换 AgentPlan.md>)** —— 内部切换：pendingPlan 如何在不停作业的前提下替换正在运行的 plan。覆盖 AgentPlan 内容与各消费组件的生效 scope、切换时机（与 checkpoint barrier 对齐）、drain 语义、一致性契约、状态与快照、恢复流程、按类列出的 Proposed Changes。

## 一图总览

```
Part.1 更新链路                              │ Part.2 数据流切换
                                            │
用户(Java/Python 客户端)                     │
  │ POST 完整 AgentPlan JSON                 │
  ▼                                         │
JM REST → PlanUpdateCoordinator(两阶段)      │      AEO subtask（每个并行度）
  · 静态校验(失败同步拒绝)                     │
  · stage:planVersion 铸造,暂不发送           │
  · cp K 持久化 {active, staged}              │
  · K 完成 → announce ───────────────────────┼──►  handleOperatorEvent
  · 重启后重宣 staged(若存在)                │      · 静态校验+准备(不进入 CPython)
                                            │      · 成为 pendingPlan
                          截面: PlanUpdateEvent      ▼
                                            │    checkpoint K+1 的 prepareSnapshotPreBarrier
                                            │      ① drain:输入集合封闭,等在飞事件链按旧 plan 结束
                                            │      ② quiesce/关闭旧资源,清理旧 Python 用户模块
                                            │      ③ 挂载新 artifact,创建 runtime,预解析 action 入口
                                            │      ④ switch 后快照记录新 plan(union state)
                                            │    barrier 之后的数据按新 plan 执行
```

## 关键决策速览

| 决策点 | 结论 | 详见 |
| --- | --- | --- |
| 更新通道 | OperatorCoordinator 广播（Flink 原生：同步校验反馈、单点定序、事件与 cp 的 epoch 一致性、coordinator 状态随 cp 恢复 + backfill） | Part.1 |
| 提交物 | 完整 AgentPlan JSON 全量快照；planId = sha256 内容身份，回滚 = 重发旧 JSON | Part.1 |
| 生效时点 | 正常路径需要两个成功的 checkpoint：第一个 cp 持久化决定，全部 subtask 静态 prepare 成功后在第二个 cp 的 barrier 同点切换。cp abort、drain 或 reload 变慢时，墙钟延迟无上界 | Part.2 |
| 在飞数据 | 切换前 drain：barrier 后不再接纳新输入，已进入算子的事件链继续按旧 plan 执行；该输入集合已封闭，但用户事件链本身不保证终止。drain 与 Python reload 均无本地超时 | Part.2 |
| 运行时模型 | 单版本 + 单 pending 两槽位，无多版本共存；更新到达时只做静态校验和 Java 侧准备，Python artifact 完整重载放在激活 barrier 的 drain 之后 | Part.2 |
| 失败恢复 | coordinator 持久化 {active, staged}，被打断的切换恢复后自动续上；subtask 从 union state 恢复 currentPlan。只有全部 subtask 静态 prepare 成功时，完成的切换 cp 中各条目才保证同版本 | Part.1 + Part.2 |
| 配置边界 | 作业级配置钉死在提交时刻，`dynamic-plan.python-runtime.path` 也不可动态修改；只有 Java/Python artifact 的 path + sha256 随 plan 更新 | Part.2 |
| Python 代码重载 | 配置 artifact 时从 zip/whl 内容提取新旧顶层模块根，按 root 整棵清理 `sys.modules`、import/function cache，再加载并预解析 V2 action 入口；未声明 artifact 的 plan 不贡献新清理根，不承诺代码热更新 | Part.2 |
| 部署边界 | coordinated AEO 禁止 chaining、使用专用 slot-sharing group；生产要求每个 TM 只有 1 个 slot，且该 JVM/CPython 进程只有一个 AEO Python owner。专用 group 会额外占用 slot；协议还要求 `execution.checkpointing.max-concurrent-checkpoints = 1` | Part.2 |
| 一致性 | 全部 subtask 通过静态 prepare 时，保证单链版本纯净 + 全局同 cp 切换；MVP 尚无 prepare nack，环境不一致可造成分叉；另有激活 cp abort 的跨 attempt 重放差异 | Part.2 |

## 现状

方案已在分支 `dynamic-plan-v2` 实现。多版本共存的替代方案曾以 POC 形式验证（分支 `dynamic-plan-coordinator`），本方案复用其中的 coordinator 与 artifact 校验能力,但 Python 切换改为 activation barrier drain 后的单 runtime 完整重载,不再依赖多个 interpreter 的模块隔离。
