# AgentPlan动态更新设计文档总览

## 动机

Flink Agents 作业通常是长期运行的。作业提交后，Agent 会持续消费业务事件，并依靠 keyed state、checkpoint 和 durable execution 保证一致性与恢复。

当前 `AgentPlan` 在作业提交时固定下来。`AgentPlan` 包含 action 编排、事件路由、prompt/resource/tool/model 配置。改一句 prompt、换一个 model、重定向一个 tool，或者调整 action 连接关系，都需要重新提交作业。

这对生产环境的 Agent 成本很高：

*   重启会中断正在处理的事件链。

*   变更链路长，策略调整不够快。

*   已有 keyed state 和 durable execution 上下文需要随作业恢复，操作风险高。

*   对 prompt、resource、tool/model 这类策略配置来说，重提作业太重。

本设计聚焦一个核心问题：**运行中的 AEO 如何在不重启作业的情况下，从旧 AgentPlan 原地切换到新 AgentPlan，并在失败恢复后仍然用对 plan。**

## 拆分方式与截面

整个问题在一个清晰的截面处一分为二：**"新 plan 如何送达每个 AEO subtask"** 和 **"subtask 拿到新 plan 后如何切换"**。截面本身就是一条消息加一个约定——coordinator 广播的 `PlanUpdateEvent(seq, planId, planJson)`，以及 subtask 侧"收到后先校验准备、成为 pendingPlan、等下一个 checkpoint 生效"的处理契约。截面之前几乎全是 Flink 原生组件（REST、OperatorCoordinator、SubtaskGateway），与 Agent 语义无关；截面之后全是 AEO 内部的事，与传输通道无关。两侧可以独立设计、独立演进，因此各成一篇：

*   **[Part.1 AgentPlan更新链路设计](<Part.1 AgentPlan更新链路设计.md>)** —— 外部链路：用户手里的一份新 AgentPlan JSON，如何变成每个 subtask 内存中的 pendingPlan。覆盖通道选型（为什么是 OperatorCoordinator 广播而不是重提作业/broadcast stream/外部存储轮询）、Java 与纯 Python 客户端、投递与定序语义、"接受 ≠ 生效"的响应语义、两层校验、失败点走查。

*   **[Part.2 数据流切换 AgentPlan](<Part.2 切换 AgentPlan.md>)** —— 内部切换：pendingPlan 如何在不停作业的前提下替换正在运行的 plan。覆盖 AgentPlan 内容与各消费组件的生效 scope、切换时机（与 checkpoint barrier 对齐）、drain 语义、一致性契约、状态与快照、恢复流程、按类列出的 Proposed Changes。

## 一图总览

```
Part.1 更新链路                              │ Part.2 数据流切换
                                            │
用户(Java/Python 客户端)                     │
  │ POST 完整 AgentPlan JSON                 │
  ▼                                         │
JM REST → PlanUpdateCoordinator(两阶段)      │      AEO subtask（每个并行度）
  · 静态校验(失败同步拒绝)                     │
  · stage:planVersion 铸造,暂不发送           │
  · cp K 持久化 {active, staged}              │
  · K 完成 → announce ───────────────────────┼──►  handleOperatorEvent
  · 重启后重宣 staged(若存在)                │      · 校验+准备(classloader/Python runtime)
                                            │      · 成为 pendingPlan
                          截面: PlanUpdateEvent      ▼
                                            │    checkpoint K+1 的 prepareSnapshotPreBarrier
                                            │      ① drain:在飞事件链按旧 plan 跑完
                                            │      ② switch:换引用,关旧资源(全体同一 cp)
                                            │      ③ 快照记录切换后的 plan(union state)
                                            │    barrier 之后的数据按新 plan 执行
```

## 关键决策速览

| 决策点 | 结论 | 详见 |
| --- | --- | --- |
| 更新通道 | OperatorCoordinator 广播（Flink 原生：同步校验反馈、单点定序、事件与 cp 的 epoch 一致性、coordinator 状态随 cp 恢复 + backfill） | Part.1 |
| 提交物 | 完整 AgentPlan JSON 全量快照；planId = sha256 内容身份，回滚 = 重发旧 JSON | Part.1 |
| 生效时点 | 提交后第二个 checkpoint 的 barrier（两阶段：第一个 cp 持久化决定，第二个 cp 全体同点切换），cp 前后执行不同 plan，快照里存的就是切换后的 plan | Part.2 |
| 在飞数据 | 切换前 drain：每条数据及其整条事件链都按接收它时的 plan 执行完；drain 无本地超时，拖慢乃至超时的 checkpoint 属预期内，由用户配置 cp timeout 覆盖 | Part.2 |
| 运行时模型 | 单版本 + 单 pending 两槽位，无多版本共存；校验和重量级准备前移到更新到达时（唯一有意偏差，文档内有说明与回退路径） | Part.2 |
| 失败恢复 | coordinator 持久化 {active, staged}，被打断的切换恢复后自动续上；subtask 从 union state 恢复 currentPlan（两阶段下条目必然一致） | Part.1 + Part.2 |
| 配置边界 | 作业级配置钉死在提交时刻，更新只带 actions/路由/resources（例外：`dynamic-plan.*` 键） | Part.2 |
| 一致性 | 单链版本纯净 + 全局同 cp 切换（两阶段因果链保证）；残余窗口仅激活 cp abort 的重放差异，MVP 容忍 | Part.2 |

## 现状

方案已在分支 `dynamic-plan-v2` 完整实现并验证：runtime 全量单测、切换语义专项测试（cp 前旧 plan/cp 后新 plan、drain、恢复、幂等回填等 8 项）、MiniCluster 上 Java 与纯 Python REST 客户端的端到端切换均通过。多版本共存的替代方案曾以 POC 形式完整验证（分支 `dynamic-plan-coordinator`），其 coordinator、artifact 校验、Python 多 runtime 等组件被本方案直接复用。
